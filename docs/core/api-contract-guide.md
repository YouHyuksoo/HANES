# API 계약 가이드

## 목적

신규 프로젝트에서 백엔드 API를 설계할 때 경로, DTO, 응답 규칙을 일관되게 유지하기 위한 기준이다.

## 경로 규칙

1. 경로는 도메인 접두사로 시작한다. 예: `/material/*`, `/production/*`
2. 복수형 또는 업무 단위 경로를 사용하되 한 프로젝트 안에서 혼용하지 않는다.
3. 상태 전이 API는 명령 의미가 드러나는 하위 경로를 사용한다. 예: `POST /production/job-orders/:id/confirm`

## 요청 규칙

1. 필터 DTO와 생성/수정 DTO를 분리한다.
2. 날짜 필터는 문자열 포맷보다 범위 값으로 받는다.
3. 페이징 목록은 `page`, `limit`, `sort`를 기본 축으로 둔다.

## 응답 규칙

1. 목록 API는 데이터와 총 건수를 분리해 제공한다.
2. 상세 API는 화면이 직접 쓰는 필드를 우선 제공한다.
3. 상태값과 코드값은 원시값만 던지지 말고 화면에서 해석 가능한 최소 맥락을 함께 준다.

## 에러 규칙

1. 상태 전이 실패는 검증 실패 사유를 명확히 반환한다.
2. 중복, 미존재, 불가 상태는 서로 다른 메시지와 상태 코드로 구분한다.
3. UI가 바로 보여줄 수 있는 수준의 에러 메시지를 유지한다.

## 좋은 예시

- `POST /production/job-orders/:id/confirm`처럼 상태 전이 의미가 드러나는 경로
- 목록 응답에서 데이터와 총 건수를 분리한 구조

## HANES 코드 예시

- 앱 모듈 라우트 조립 기준: `apps/backend/src/app.module.ts`
- 출하 API 그룹 예시: `apps/backend/src/modules/shipping`
- 공통 코드 API 예시: `apps/backend/src/modules/master/controllers/com-code.controller.ts`

## 나쁜 예시

- `/doAction`, `/changeStatus`처럼 업무 의미가 불명확한 경로
- 상세 API가 화면이 안 쓰는 내부 필드까지 무차별 반환하는 구조

## 적용 체크리스트

1. 도메인 접두사가 명확한가
2. 필터 DTO와 생성/수정 DTO가 분리됐는가
3. 목록 응답에 총 건수와 정렬 기준이 명확한가
4. 상태 전이 실패 메시지가 UI 친화적인가

## AI 입력 순서에서의 역할

- 도메인 블루프린트 다음에 읽는다.
- API 표면을 생성할 때 직접 참조하는 문서다.
