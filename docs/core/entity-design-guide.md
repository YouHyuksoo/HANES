# 엔티티 설계 가이드

## 목적

Oracle + TypeORM 기준에서 신규 엔티티를 설계할 때 필요한 명명 규칙, 키 설계 원칙, 인덱스 원칙을 정리한다.

## 기본 원칙

1. 엔티티는 현재 상태와 이력을 구분해 설계한다.
2. 상태 컬럼은 의미 있는 업무 상태값을 사용한다.
3. 공통 컬럼은 프로젝트 전반에서 일관되게 유지한다.

## 키 설계

1. 자연키가 명확하면 우선 검토한다.
2. 복합키가 업무적으로 자연스러우면 허용한다.
3. 채번 PK를 쓰더라도 업무 식별 컬럼을 별도로 둔다.

## 컬럼 설계

1. DB 컬럼명은 명시적으로 매핑한다.
2. 회사와 공장 스코프가 필요한 테이블에는 이를 기본 컬럼으로 둔다.
3. 날짜와 시간은 검색 패턴을 고려해 분리 또는 일관된 타입으로 설계한다.

## 이력 테이블

1. 현재 재고와 트랜잭션 이력을 분리한다.
2. 상태 테이블과 이벤트 로그를 하나로 억지로 합치지 않는다.
3. 취소와 복원 로직이 필요하면 원본 참조 키를 남긴다.

## 인덱스 원칙

1. 자주 쓰는 필터 조합 기준으로 복합 인덱스를 설계한다.
2. 함수 기반 검색이 필요하면 설계 단계에서 별도 보조 전략을 마련한다.
3. 목록, 집계, 상태 전이용 인덱스를 구분해 생각한다.

## 좋은 예시

- 현재 재고 테이블과 트랜잭션 이력 테이블을 분리하는 구조
- 회사, 공장, 상태, 날짜 조합을 기준으로 복합 인덱스를 설계하는 방식

## HANES 코드 예시

- 공통 코드 엔티티: `apps/backend/src/entities/com-code.entity.ts`
- 자재/재고 엔티티 군: `apps/backend/src/entities/*.entity.ts`
- 생산 실적과 제품 재고 연결 예시: `apps/backend/src/modules/production/services/prod-result.service.ts`

## 나쁜 예시

- 모든 테이블에 의미 없는 단일 증가 PK만 두고 업무 식별 컬럼을 별도로 관리하지 않는 구조
- 검색 패턴을 모른 채 단일 컬럼 인덱스만 나열하는 방식

## 적용 체크리스트

1. 엔티티가 현재 상태와 이력을 구분하는가
2. 키 전략이 업무 식별과 일치하는가
3. 회사/공장 스코프가 필요한 엔티티에 공통 컬럼이 있는가
4. 인덱스가 실제 조회 패턴과 맞는가
