# UI 화면 패턴

## 목적

신규 프로젝트에서 화면을 설계할 때 반복 가능한 구조를 제공한다.
현재 HANES 화면 패턴을 일반화한 규격 문서다.

## 기본 화면 구조

1. 페이지 헤더
2. 필요 시 요약 카드
3. 필터와 목록
4. 상세 또는 입력 모달

## 목록 화면 패턴

1. 상단에 제목과 주요 액션을 둔다.
2. 필터와 데이터 그리드는 같은 작업 맥락에 둔다.
3. 재사용 가능한 필터 컴포넌트를 우선한다.
4. 상태값은 배지 또는 공통 코드 렌더러로 표시한다.

## 상세 및 입력 패턴

1. 등록과 수정은 가능한 한 같은 폼 구조를 재사용한다.
2. 조회 전용 필드와 입력 필드를 시각적으로 구분한다.
3. 저장 전 검증 메시지는 필드 단위와 폼 단위를 나눠 보여준다.

## PDA 화면 패턴

1. 스캔 입력 흐름을 가장 앞에 둔다.
2. 키보드 입력보다 바코드 입력을 우선한다.
3. 실패 메시지는 즉시 보여주고 다음 스캔 흐름을 막지 않도록 설계한다.

## 공통 코드 패턴

1. 상태와 분류 텍스트는 하드코딩하지 않는다.
2. 공통 코드 배지, 셀렉트, 라벨 훅을 제공한다.
3. 번역 키와 DB 코드명 사이의 폴백 규칙을 문서화한다.

## 좋은 예시

- 페이지 헤더, 필터, 목록, 상세 모달의 4단 구조를 반복 가능한 화면 기본형으로 쓰는 방식
- PDA 화면에서 스캔 입력과 즉시 피드백을 가장 앞에 두는 방식

## HANES 코드 예시

- 공통 코드 배지: `apps/frontend/src/components/ui/ComCodeBadge.tsx`
- 공통 코드 셀렉트: `apps/frontend/src/components/shared/ComCodeSelect.tsx`
- 공통 코드 훅: `apps/frontend/src/hooks/useComCode.ts`

## 나쁜 예시

- 화면마다 다른 필터 배치를 임의로 쓰는 방식
- 상태값 텍스트와 배경색을 각 페이지에서 직접 정의하는 방식

## 적용 체크리스트

1. 화면이 헤더, 목록, 상세의 기본 구조를 따르는가
2. 공통 필터와 공통 상태 렌더러를 재사용하는가
3. PDA 흐름이 스캔 우선으로 설계됐는가
4. 반복되는 UI가 공통 컴포넌트로 승격됐는가

## AI 입력 순서에서의 역할

- 엔티티 설계 가이드 다음에 읽는다.
- 화면과 컴포넌트 구조를 생성할 때 기준이 되는 문서다.
