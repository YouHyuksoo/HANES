# 안티패턴

## 목적

HANES 코드 리뷰에서 반복적으로 드러난 문제를 금지 규칙 수준으로 정리한다.
신규 프로젝트에서 같은 실수를 다시 만들지 않도록 하기 위한 문서다.

## SQL 안티패턴

1. 목록 조회 후 루프 안 `findOne`
2. 루프 안 `count`, `update`, `save`
3. 컬럼 쪽 `TRUNC`, `TO_CHAR`, `UPPER`, `LOWER` 남용
4. 전량 조회 후 메모리 필터링
5. 느슨한 연결 키로 취소와 복원 처리

## 도메인 안티패턴

1. 출하지시와 출하를 문서상 하나로 적고 코드에선 둘로 운영
2. 생산실적 완료와 작업지시 완료를 같은 것으로 가정
3. 현재 재고와 이력 재고를 구분하지 않음

## UI 안티패턴

1. 상태 텍스트 하드코딩
2. 같은 필터 UI를 화면마다 새로 작성
3. PDA 화면에서 스캔보다 수기 입력을 우선 배치

## 문서 안티패턴

1. 코드와 다른 상태값을 문서에 유지
2. 생성물과 기준 문서를 같은 폴더에 둠
3. 설명 문서를 규격 문서처럼 오용

## 좋은 예시

- 문서와 코드의 상태값을 함께 갱신하는 습관
- 생성물은 `exports/`, 기준 문서는 `docs/`로 분리하는 구조

## HANES 코드 검토 포인트

- N+1과 처리성 병목이 자주 생기는 축: `apps/backend/src/modules/material/services/physical-inv.service.ts`
- 생산 완료와 재고 반영 결합 축: `apps/backend/src/modules/production/services/prod-result.service.ts`
- 출하 상태 전이와 재고 차감 축: `apps/backend/src/modules/shipping/services/shipment.service.ts`

## 나쁜 예시

- 옛 요구사항 문서를 기준 문서처럼 남겨 두는 방식
- 성능 병목이 반복되는데도 N+1과 함수 기반 검색을 그대로 복제하는 방식

## 적용 체크리스트

1. 새 설계에 루프 안 조회가 없는가
2. 컬럼 함수 기반 검색을 피했는가
3. 현재 상태와 이력이 혼재되지 않았는가
4. 기준 문서와 산출물이 섞이지 않았는가

## AI 입력 순서에서의 역할

- 마지막에 읽는다.
- 생성 전에 금지 패턴을 다시 상기시키는 문서다.
