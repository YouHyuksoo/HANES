# 아키텍처 원칙

## 목적

이 문서는 HANES에서 검증된 구조를 다른 신규 프로젝트에 재사용할 수 있도록 아키텍처 원칙을 규격 형태로 정리한 문서다.
AI가 신규 프로젝트를 설계하거나 코드를 생성할 때 기본 설계 기준으로 사용한다.

## 기본 원칙

1. 모듈은 업무 도메인 기준으로 나눈다.
2. 컨트롤러는 입출력과 권한 처리만 담당한다.
3. 서비스는 비즈니스 규칙과 트랜잭션 경계를 담당한다.
4. 엔티티는 저장 구조를 표현하되 업무 흐름까지 품지 않는다.
5. 조회 문서와 규격 문서는 분리하되, 기준 원본은 항상 코드다.

## 권장 계층

### 프론트엔드

- 화면
- 공통 컴포넌트
- 훅
- API 클라이언트
- 타입 및 화면 전용 유틸

### 백엔드

- Controller
- DTO
- Service
- Repository 또는 TypeORM Entity 접근
- Oracle 보조 호출

## 모듈 경계 규칙

1. 모듈 이름은 업무 용어를 따른다. 예: `material`, `production`, `quality`, `shipping`
2. 한 모듈은 하나의 주요 업무 흐름을 책임진다.
3. 다른 모듈의 내부 서비스에 직접 의존하지 말고, 필요한 경우 명시적 서비스 인터페이스나 공용 조회 계층을 둔다.
4. `master`는 참조 데이터 제공자 역할을 하되 업무 흐름의 소유자가 되지 않는다.

## 상태 흐름 원칙

1. 상태값은 문자열 enum 수준으로 명시한다.
2. 상태 전이는 서비스 메서드에서만 수행한다.
3. 상태 전이 전 검증 규칙을 같은 메서드 안에서 처리한다.
4. 상태 전이 문서와 실제 코드 값은 항상 동일해야 한다.

## 트랜잭션 원칙

1. 재고 차감, 실적 완료, 입고 확정, 출하 확정 같은 처리성 로직은 서비스 단에서 트랜잭션을 가진다.
2. 트랜잭션 안에서 루프별 DB 왕복을 늘리지 않는다.
3. 재고 변경과 이력 기록은 분리하지 말고 한 트랜잭션 안에서 완료한다.

## AI 생성 지침

1. 새 기능은 기존 모듈에 자연스럽게 들어가면 새 모듈을 만들지 않는다.
2. 새 모듈이 필요하면 도메인 용어, 엔드포인트 접두사, 상태값, 주요 엔티티를 먼저 정의한다.
3. 구현 전에 반드시 워크플로우와 상태 전이를 먼저 문서화한다.

## 좋은 예시

- `material`, `production`, `quality`, `shipping`처럼 업무 경계가 이름과 책임에 드러나는 모듈 구성
- 상태 전이 메서드가 서비스에 모여 있고, 컨트롤러는 요청 진입점 역할만 하는 구조

## HANES 코드 예시

- 루트 모듈 조립: `apps/backend/src/app.module.ts`
- 생산 상태 전이와 업무 로직 집중 예시: `apps/backend/src/modules/production/services/job-order.service.ts`
- 출하 상태 전이 예시: `apps/backend/src/modules/shipping/services/shipment.service.ts`

## 나쁜 예시

- 컨트롤러에서 직접 트랜잭션과 재고 차감을 수행하는 구조
- `common`, `shared-service`, `misc`처럼 책임이 불명확한 모듈 남용

## 적용 체크리스트

1. 새 기능의 도메인 소유 모듈이 명확한가
2. 컨트롤러에 업무 로직이 새어 나오지 않았는가
3. 상태 전이와 검증이 서비스에서 함께 처리되는가
4. 트랜잭션 경계가 서비스 메서드 단위로 정의되는가

## AI 입력 순서에서의 역할

- 신규 프로젝트 주입 시 가장 먼저 읽히는 문서다.
- 이후 `implementation-rules.md`, `domain-blueprints.md`로 내려간다.
