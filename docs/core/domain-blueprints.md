# 도메인 블루프린트

## 목적

신규 프로젝트에서 재사용 가능한 도메인 템플릿을 정리한다.
각 블루프린트는 엔티티 축, 상태 축, 핵심 처리 축을 함께 제공한다.

## 자재 도메인

### 핵심 엔티티

- 입하
- LOT
- 검사 로그
- 입고
- 현재 재고
- 재고 이력

### 기본 흐름

1. 입하 등록
2. LOT 생성
3. 검사 판정
4. 적합 LOT 입고
5. 재고 반영

### 설계 포인트

- 검사 결과와 입고 가능 여부를 분리한다.
- 현재 재고와 이력 재고를 함께 유지한다.
- 취소와 복원 시 연결 키가 느슨해지지 않도록 한다.

## 생산 도메인

### 핵심 엔티티

- 생산계획
- 작업지시
- 생산실적
- 제품 재고
- 제품 이력

### 기본 흐름

1. 계획 생성
2. 계획 확정
3. 작업지시 발행
4. 작업 시작
5. 실적 등록
6. 완료 및 재고 반영

### 설계 포인트

- 계획, 작업, 실적 상태를 분리한다.
- 자동출고와 생산완료를 한 트랜잭션 축으로 본다.
- WIP와 FG를 같은 테이블에 섞을지 분리할지 초기에 결정한다.

## 품질 도메인

### 핵심 엔티티

- 검사결과
- 불량로그
- 추적 로그
- OQC
- SPC 계열 데이터

### 기본 흐름

1. 검사 결과 등록
2. 불합격 처리
3. 후속 조치
4. 추적성 조회

### 설계 포인트

- 검사와 불량을 동일 테이블에 과하게 합치지 않는다.
- 추적성은 생산, 자재, 설비, 작업자를 연결해야 한다.

## 출하 도메인

### 핵심 엔티티

- 고객주문
- 출하지시
- 출하
- 박스
- 팔레트
- FG 라벨

### 기본 흐름

1. 주문 등록
2. 출하지시 생성
3. 포장
4. 적재
5. 출하
6. 배송 완료

### 설계 포인트

- 출하지시와 실출하를 같은 상태머신으로 볼지 분리할지 먼저 정한다.
- 출하와 재고 차감은 같은 원자 단위로 설계한다.

## 좋은 예시

- 자재에서 `입하 -> 검사 -> 입고 -> 현재 재고 + 이력 재고`처럼 현재 상태와 이벤트 이력을 함께 설계
- 생산에서 `계획`, `작업지시`, `실적`의 상태를 분리해 흐름을 명확히 표현

## HANES 코드 예시

- 자재 실사 흐름: `apps/backend/src/modules/material/services/physical-inv.service.ts`
- 생산 흐름: `apps/backend/src/modules/production/services/job-order.service.ts`, `apps/backend/src/modules/production/services/prod-result.service.ts`
- 출하 흐름: `apps/backend/src/modules/shipping/services/shipment.service.ts`

## 나쁜 예시

- 품질과 불량을 한 테이블에 몰아 넣어 후속 처리 분기가 복잡해지는 구조
- 출하지시와 출하를 문서에서는 하나로 적고 코드에서는 둘로 운영하는 구조

## 적용 체크리스트

1. 각 도메인에 핵심 엔티티, 상태, 이벤트 이력이 따로 정의됐는가
2. 취소와 복원 시 원본 참조 키를 추적할 수 있는가
3. 현재 재고와 이력 재고가 분리되는가
4. 출하와 재고 차감이 같은 트랜잭션 축으로 설계되는가

## AI 입력 순서에서의 역할

- 구현 규칙 다음에 읽는다.
- 도메인 모델을 빠르게 세우는 템플릿 문서다.
