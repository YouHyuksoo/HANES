# 구현 규칙

## 목적

AI가 신규 프로젝트 또는 후속 기능을 구현할 때 지켜야 하는 실무 규칙을 정리한다.

## 백엔드 규칙

1. NestJS 모듈 구조를 유지한다.
2. Oracle을 기준 DB로 본다.
3. 엔티티 컬럼명은 실제 DB 컬럼과 명시적으로 매핑한다.
4. 복합키와 자연키를 우선 검토하고, 무의미한 자동 증가 키 남용을 피한다.
5. 상태 변경 메서드는 서비스에서 이름이 드러나게 작성한다. 예: `confirm`, `complete`, `cancel`

## SQL 규칙

1. 루프 안 `findOne`, `count`, `update`를 기본 구현으로 두지 않는다.
2. `TRUNC`, `TO_CHAR`, `UPPER`, `LOWER`를 컬럼 쪽에 무분별하게 적용하지 않는다.
3. 목록 조회는 기본적으로 페이징을 고려한다.
4. 검색은 배치 조회, `IN`, `JOIN`, 범위 조건을 우선 검토한다.
5. 실행 계획 검증이 필요한 쿼리는 서비스 코드와 함께 기록한다.

## 프론트엔드 규칙

1. 목록 화면은 필터, 테이블, 상세 액션의 세 축으로 설계한다.
2. 상태값 표시는 하드코딩하지 않고 공통 코드나 상태 매퍼를 사용한다.
3. PDA 화면은 입력 속도와 스캔 흐름을 우선한다.
4. 공통 UI는 반복되기 전에 컴포넌트로 승격한다.

## 문서 규칙

1. 문서는 코드에서 검증 가능한 사실만 기준 문서에 남긴다.
2. 파일명은 `lowercase-kebab-case`를 사용한다.
3. AI 주입용 문서는 설명보다 규칙과 예외를 우선 적는다.

## 좋은 예시

- 상태 변경 메서드를 `confirm`, `complete`, `cancel`처럼 드러나는 이름으로 분리
- 검색 조건을 날짜 범위, 배치 조회, `JOIN` 중심으로 설계
- 화면에서 상태 텍스트를 공통 코드와 배지 컴포넌트로 렌더링

## HANES 코드 예시

- 생산 완료 처리와 재고 반영: `apps/backend/src/modules/production/services/prod-result.service.ts`
- 자재 재고 서비스: `apps/backend/src/modules/material/services/mat-stock.service.ts`
- 공통 코드 프론트 훅: `apps/frontend/src/hooks/useComCode.ts`

## 나쁜 예시

- 루프 안 `findOne`, `count`, `save`
- 컬럼 쪽 `TRUNC`, `TO_CHAR`, `UPPER`, `LOWER` 남용
- 화면에서 상태 문자열과 색상을 직접 하드코딩

## 적용 체크리스트

1. 목록성 SQL에 N+1 패턴이 없는가
2. 날짜 필터가 함수가 아니라 범위 조건인가
3. 공통 UI로 올릴 반복 코드가 방치되지 않았는가
4. 문서가 설명서가 아니라 규칙서 형태를 유지하는가

## AI 입력 순서에서의 역할

- 아키텍처 원칙 다음에 읽는다.
- 구현 세부 규칙과 금지사항을 AI에게 주입하는 문서다.
